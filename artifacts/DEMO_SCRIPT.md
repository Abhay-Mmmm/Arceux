# Arceux — 5-Minute Demo Script

A step-by-step guide for presenting Arceux to a technical audience. Each minute has a talk track and specific things to point at on screen.

---

## Before You Begin

Start the system at least 90 seconds before the demo begins so River ML has time to warm up and alerts start accumulating.

```bash
# Terminal 1
cd server && python main.py

# Terminal 2
cd client && npm run dev
```

Open http://localhost:5173 in a browser, full-screen. Confirm the "WS Live" indicator in the top-right corner of the dashboard is green.

---

## Minute 1 — System Overview

**What to point at:** The Dashboard page on load.

**What to say:**

> "This is Arceux — an AI-native Security Operations Center built for financial institutions. What you're looking at is the live dashboard. Every component here is driven by real data from the backend.
>
> The six waveforms at the top are the System Heartbeat — each one represents a backend component: Log Collector, Threat Intelligence, SIEM Engine, Alert Pipeline, Analytics Engine, and Database. The waveform shape is derived from actual latency history coming from the server, not generated client-side.
>
> The green 'WS Live' indicator tells you we have a WebSocket connection open to the server. Alerts, metrics, and compliance updates will arrive on this channel without the page ever needing to poll."

**Tip:** If the WS indicator is gray, refresh the page. The frontend reconnects automatically but the indicator may lag on first load.

## Minute 2 — Live Detection

**What to point at:** High Priority Alerts feed (right side of Dashboard), the Threat Severity bar chart (center).

**What to say:**

> "Watch the High Priority Alerts feed on the right. These are arriving via WebSocket push — the instant the backend detection engine fires and the CrewAI agents produce an alert, it appears here, without a page refresh.
>
> The detection stack has three layers. First, six deterministic rule-based detectors: brute force (>5 failed logins in 2 minutes), impossible travel (same user, two countries, 10-minute window), insider threat (privilege escalation then data download), and others. These fire synchronously on every ingested log.
>
> When no rule fires, the second layer kicks in — River HalfSpaceTrees, an online ML model that learns continuously and scores behavioral anomalies against a per-user-cohort baseline. No retraining, no batch jobs.
>
> The third layer is a NetworkX graph detector. It builds a rolling temporal graph of IP-to-user and user-to-asset relationships, and checks for structural patterns like lateral movement — one IP authenticating as three different users in a five-minute window — that no single-event rule could catch.
>
> The bar chart in the center shows the live distribution across CRITICAL, HIGH, MEDIUM, and LOW. It updates via WebSocket push every time a new alert arrives."

## Minute 3 — Agent Pipeline

**What to point at:** Navigate to the Agent Insights page (sidebar nav).

**What to say:**

> "This is the Agent Insights page. The six boxes across the middle are the CrewAI agents: Orchestrator, Alert Handler, Threat Analyzer, Root Cause, Compliance, and Response Automation. Each box shows live status — idle, running, completed, or failed.
>
> Watch the activity bar at the very top of the page."

**Point at the activity bar — it should show amber/pulsing if a pipeline just ran, or green checkmark if one completed recently.**

> "That bar is driven entirely by WebSocket events — no additional API calls. When an agent is running, you see a pulsing amber indicator with the signal type and which agent is analyzing. When the pipeline completes, it switches to green and shows you how many agents ran and the elapsed time.
>
> Here's the key design decision: not all six agents run on every signal. The pipeline uses signal-type routing. A Brute Force detection runs only three agents — Alert Handler, Threat Analyzer, and Response Automation — which cuts token usage by about 50%. An Insider Threat is the most dangerous pattern so it runs all six.
>
> Click any agent card."

**Click on an agent card that has a 'completed' status.**

> "This is the last execution trace — the actual output lines from that agent's last Groq call. Task count, average execution time. Each agent also has its own dedicated Groq API key so its rate limit is isolated from the others."

## Minute 4 — Investigation

**What to point at:** Navigate to the Alerts page.

**What to say:**

> "This is the Alerts page. Every alert generated by the detection and agent pipeline shows up here with severity, confidence score, user, asset, and status."

**Point at the filter bar and confidence column.**

> "Confidence is dynamic: for River ML alerts, it's the raw anomaly score multiplied by 100. For rule-based alerts, it's derived from severity. Nothing is hardcoded.
>
> Click on a CRITICAL alert to open the detail panel."

**Click a CRITICAL alert — the slide-out panel opens on the right.**

> "The detail panel shows the full AI agent trace — each line is what a specific agent concluded about this incident. So you can see the Alert Handler's triage finding, then the Threat Analyzer's MITRE ATT&CK mapping, then the Compliance Agent's regulatory assessment.
>
> Click Take Ownership."

**Click Take Ownership — watch the status badge update immediately.**

> "That's an optimistic UI update — the status changes locally before the PATCH request to the backend even returns, so the analyst doesn't wait. The backend confirms asynchronously.
>
> Now click one of the Execute buttons — Block IP or Reset Credentials."

**Click Block IP (or Reset Credentials).**

> "The button shows a spinner, then 'Executed'. The backend logs the action and transitions the alert status to 'investigating' automatically. Once executed, the button is permanently disabled — you can't double-execute even if the page re-polls."

## Minute 5 — Compliance + Chatbot

**What to point at:** Navigate back to the Dashboard.

**What to say:**

> "Back to the Dashboard. Look at the Compliance card in the bottom-right."

**Point at the Compliance card.**

> "Every row here is driven by live data from the backend — no hardcoded values. IRDAI requires financial institutions to report critical incidents within 6 hours. If there's an unresolved CRITICAL alert, you see the countdown timer ticking down in real-time, client-side, updated every minute. Under 60 minutes it turns red. Under 30 minutes it starts pulsing.
>
> GDPR tracks data-breach signals — Insider Threat and Anomalous Access — and gives a 72-hour reporting window. SOC 2 scales with the total unresolved alert count. All five regulations recompute instantly via WebSocket push when any alert status changes."

**Now click in the chatbot input and type:** `What users are most at risk right now?`

> "The chatbot is Arceux AI — a senior SOC analyst persona running on Groq's llama-3.1-8b-instant. Every call injects the live system state: the 10 most recent alerts sorted by severity, agent pipeline status, and severity counts. This isn't a generic LLM — it knows what's happening in this environment right now."

**Wait for the response — it will reference actual alert data.**

> "Notice it named specific users and threat types from the live alert list. Multi-turn conversation is supported — you can follow up with 'tell me more about that user' and it will remember what it just said.
>
> To summarize the architecture story: synthetic logs from the generator hit the detection engine — three independent layers — signals route to the CrewAI pipeline — agents call Groq — alerts are stored and broadcast over WebSocket — the frontend receives them in under a second. The whole path from log to analyst dashboard is fully automated, with compliance posture updating continuously as the threat landscape changes."

## Common Questions

**Q: Is this production-ready?**
No — it uses in-memory storage (alerts reset on restart), has no authentication, and connects to synthetic logs only. It's a prototype demonstrating the architecture and AI integration patterns.

**Q: What happens if Groq is unavailable or rate-limited?**
The agent pipeline falls back to a simulated trace with "Alert generated by detection rule" as the explanation. The chatbot falls back to data-driven text responses built from real storage data — not generic templates. The detection engine and all rule-based alerts continue operating normally since they don't call any LLM.

**Q: Why three separate Groq models / API keys?**
The chatbot uses `llama-3.1-8b-instant` (fast, cheap) while agents use `llama-3.3-70b-versatile` (high quality for deep analysis). Separate keys per agent allow each to draw from its own rate-limit pool — with separate Groq accounts, the pipeline is effectively isolated from the chatbot's traffic.

**Q: How does the graph detector find lateral movement if no single log shows it?**
The NetworkX graph accumulates edges over a rolling 5-minute window — `ip→user` edges for login events and `user→asset` edges for downloads. When the edge from one IP reaches 3 distinct users in that window, the lateral movement pattern fires. No single log event contains that information; it emerges from the graph structure.
